# Изменения функционала
## Версия iMark 1.0.3.318
### 08.11.2024
- **Проверка кодов маркировки.** Исправлена работа метода проверки КМ `GET unique_product_stamp`, используемого в веб-интерфейсе и в клиенте для проверки кода маркировки. Теперь проверка кодов маркировки через интерфейс происходит корректно.
- Обновлены **скрипты** для работы с **iMark** влияющие на механизм проверки кодов маркировки. Обновленные скрипты будут автоматически загружены установленными экземплярами **iMark** версии 1.0.3 в течение получаса после публикации. Ручных манипуляций для обновления скриптов производить не требуется.
- Увеличены внутренние интервалы ожидания соединения с **CDN** и ожидания ответа, что позволит при медленном интернете дожидаться ответа от **ГИС МТ** в отведенное время, установленное общим таймаутом в настройках. Теперь на каждый **CDN** установлено внутреннее максимальное время ожидания получения ответа в 1.5 секунд. Если за это время ответ от **CDN** не будет получен, то на оставшееся на проверку общее время будет произведено переключение на следующие **CDN**. Таким образом, на медленных соединениях можно увеличить стандартный таймаут в настройках, чтобы гарантированно дождаться ответа, а не переключаться на следующие **CDN**.
- Изменен алгоритм проверки неполных кодов маркировки, которые посылает в дополнительных запросах на проверку **Фронтол**. Теперь по **gtin** и **серийному номеру** из короткого кода находится полный код маркировки в базе **iMark** и производится проверка полного кода маркировки в **ГИС МТ**. Благодаря этому при работе в связке с **Фронтол** не обязательно отключать локальную проверку в настройках **iMark**.

- **Модуль Маркировка. Загрузка ТТН из МАРКИРОВКИ.** В форме проверки загруженного документа при создании новой карточки товара добавлен вызов скрипта **Проверка перед записью** из точки зрения.
- **Модуль Маркировка.** Добавлена возможность выгрузки документа **Приходная накладная** в документ **Выбытие**.
- **Модуль Маркировка. Документ Выбытие.** Добавлена колонка **Вес**. Колонка доступна при формировании Выбытия по ОСУ. В колонку необходимо вводить вес единицы списываемой продукции.
- **Модуль Маркировка. Документ Маркировка остатков.** Добавлена колонка Вес. При запросе остатков по ОСУ в колонку **Вес** будет записан вес продукции, числящейся в ЧЗ. Исходя из этого веса будет рассчитан вес единицы продукции и передан, при необходимости в документ Выбытие, для приведения остатка продукции в ЧЗ к учетному.
- **Модуль Маркировка. Документ Маркировка остатков.** Добавлен вариант документа Запрос остатков по ОСУ. Запрос остатков по ОСУ позволяет загрузить все товары, числящиеся в ЧЗ на объемно-сортовом учете, для проведения их инвентаризации.
- **Инвентаризация ТМЦ.** В кнопку **Пересчитать** добавлена возможность замены в документе строк с производимой продукцией на её ингредиенты. Действие позволяет упростить проведение инвентаризации полуфабрикатов и готовой продукции.
- **Документ Расходная накладная. Вариант Внутреннее перемещение ЕГАИС.** Убрана проверка цены и получения марки от контрагента.
- **Модуль ЕГАИС.** Добавлен вариант инвентаризации ЕГАИС для инвентаризации остатков разливной продукции в общепите. Описание алгоритма инвентаризации остатков разливной продукции см. в документации на модуль ЕГАИС.
- **Схемы РМК.** Добавлен скрипт, позволяющий описать дополнительные параметры отдельной строки регистрации в модуле Айтида РМК. Эти параметры определяют внешний вид и состав строки в регистрации на экране. Функционал расширяет возможности настройки модуля Айтида РМК, давая возможность расширять доступ кассира к необходимой информации о продаваемом товаре.
- **Схемы РМК.** Добавлена возможность указать интервал таймера. С указанной периодичностью будет вызываться пользовательский скрипт (**Проверка значений и прав доступа** со значением **_ИМЯПОЛЯ** равным **timer**), который сможет производить мониторинг необходимых параметров, взаимодействовать с кассиром и производить иные действия.
- **Схемы РМК.** Добавлена параметр, указывающий РМК что выбор из списков нужно производить только двойным щелчком мыши.
- **Модуль Маркировка.** Произведен переход на работу с API СУЗ 3.0
- **Работа с оборудованием. ФР.** Для ККТ Атол реализован расширенный диалог настроек параметров ККТ. В новом диалоге реализована установка дополнительных реквизитов, влияющих на пробитие чеков и работу с ФР, которые ранее устанавливались через константы. Перечень настроек сохраняется в реестр на локальном компьютере с привязкой к типу и коду оборудования. Для моделей ККТ других производителей такой же переход будет осуществлен в следующих релизах.
- **Профили ФР.** Для предприятий общественного питания по товарным группам "Упакованная вода", "Молочная продукция", "Пиво и пивные напитки" реализован функционал реализации маркированной продукции через ККТ по коду товара (по GTIN) с заполнением соответствующих отраслевых реквизитов чека.
- **Профили ФР.** Для ККТ Атол убраны все неактуальные отчеты из списка доступных отчетов.
- **Профили ФР**. Исправлены ошибка 422 (данный КМ отсутствует в таблице) при проверке КМ через ККТ.
- **Работа с оборудованием. Принтеры чеков.** Реализован механизм печати марок на сервисные принтеры (принтеры чеков) по событиям: "откладывание чека" - печатаются все ненапечатанные марки, "отмена чека" - печатается список с отмененными позициями, "удаление позиции" - печатается марка с отмененной позицией.
- **Работа с оборудованием. Принтеры чеков.** Реализован механизм фиксации признака напечатанной марки для позиции, для исключения ситуаций с повторной печатью марок на уже напечатанные ранее позиции в чеке.
- **Работа с оборудованием.** Реализована поддержка актуальной версии API при обмене с Бифит.
- **Работа с оборудованием.** Добавлена поддержка работы с терминалом оплаты Mertech через REST API. В список профилей добавлен соответствующий профиль. Теперь в связке с этим терминалом доступно получение статусов оплаты динамический QR кодов.
- **Прочее.** Добавлена функция `Сервис.РМКЗагрузитьДанныеВсехРМ()` для загрузки данных из всех РМК без вывода промежуточных результатов по каждому рабочему месту, а с выводом одного итогового сообщения. Функцию можно использовать для кнопки на панели инструментов для ручной загрузки данных.
- **Прочее.** Исправлены ошибки при загрузке данных о продажах из личного кабинета Атол Sigma, а так же при выгрузке товаров в кабинет.
- **Прочее. Проверка КМ на ККМ** - добавлена проверка принадлежности товара к фирме оборудования. Теперь при подключении более одного ФР маркированная продукция будет проверяться только если организация (код ФР) товара будет соответствовать коду ФР оборудования.
- **Прочее.** Исправлены ошибки при выгрузке документов в Атол Mark.Scan.
- **Прочее.** Исправлена ошибка, из-за которой при восстановлении отложенного чека не был определен вид операции при проверке КМ по Разрешительному режиму.
- **Прочее.** Исправлена ошибка печати нефискального чека в РМК на ККТ Штрих для прочих документов.
- **Маркировка.** Добавлен пример шаблона этикетки, формирующий DataMatrix по всем правилам GS 1. Шаблон называется **Этикетка - маркировка одежды (Полная + GS1 Datamatrix) 2024**. Шаблон может быть использован самостоятельно, а также как основа для собственных этикеток.
- **Модуль Маркировка.** Отправка постановок на кран в ГИС МТ. Теперь при отправке постановки на кран в ГИС МТ исключена передача срока годности, указанного при постановке. Данные о предельном сроке реализации после подключения к оборудованию для розлива рассчитываются автоматически в ГИС МТ по значению срока, указанного производителем. При этом, в РМК сохранена возможность указывать срок годности для внутреннего учета сроков разливаемой маркированной и немаркированной продукции. 
- **Модуль Маркировка.** **Проверка авторизации в настройках модуля.** Добавлены проверки: 
	- на регистрацию участника оборота в ГИС МТ;
	- на зарегистрированную товарную группу, выбранную в настройках модуля и вывод соответствующего сообщения при отсутствии подключенной товарной группы или отсутствии регистрации участника;
	- при наличии настроек для работы со Станцией Управления Заказами так же проверяются и связь с СУЗ для проверки корректности указанных настроек. 
- **Модуль Маркировка. Заказ кодов маркировки.** Усилен контроль по заказам кодов маркировки, которые были отправлены и зарегистрированы в ГИС МТ. Теперь в таких заказах недоступно редактирование количества. 

#### Изменения в модуле Айтида РМК.
- При вызове прочего отчета для печати в случае, если у отчета есть дополнительные параметры, эти параметры будут запрошены у кассира.
- Подбор модификаторов в чек. Для удобства выбора среди большого количества, модификаторы разбиты на группы.
- В подборе модификаторов реализована обработка признака: **Только один модификатор из группы**.
- В подборе модификаторов реализована обработка признака: **Автоматически добавлять все модификаторы к строке чека**.
- При формировании возврата по чеку, в случае оплаты исходного чека подарочной картой, эта карта автоматически добавляется в чек возврата.
- Убрано применение автоматических скидок на чек при выписке чека (возврата) на основании другого чека
- Добавлена возможность задать максимальную ширину и высоту окон выбора (товара/чека и т.п.)
- В окне не завершенного чека кнопка Отмена теперь действует иначе. Чек не удаляется полностью, а после отмены в ККТ, ПС, ЕГАИС загружается в текущую регистрацию.
> Внимание! Для правильно работы функционала, требуется обновить профили работы с ФР.
{.is-warning}
- Добавлен контроль суммы чека, в случае, если было произведено распределении суммы оплаты по видам оплаты. В случае изменения суммы чека, распределение будет удалено.
- Добавлены команды `УДАЛИТЬДОПСВОЙСТВО`, `ДОБАВИТЬДОПСВОЙСТВО`, `ОЧИСТИТЬДОПСВОЙСТВА`, позволяющие "вручную" управлять выводом информации в строке чека.
- При проверке прочитанной марки, алкогольной или КМ, в случае, если такая марка уже есть в чека, то курсор будет переведен на строку с этой маркой.
- В меню Сервис/Ещё пункт Z-отчет заменен на Закрытие смены. Применено право доступа на этот пункт.
- В форму подбора товаров, выбора, скидок, дисконтных карт, клиентов владельца карт, сотрудников добавлена возможность выводить собственные новые колонки с любой информацией. Например, в подбор товаров можно добавить колонку(и) с остатками на нескольких складах.
- В запросе параметров добавлена возможность описывать и создавать кнопки с обработкой нажатия.
- В группе кранов разливных напитков добавлены признаки, позволяющие отказаться от формирования документов **Вскрытие тары** и **Выбытие маркировки** при постановке на кран.
- Исправлена (обойдена) проблема, возникавшая при использовании драйвера платежных систем Атол и выражавшаяся в спонтанном отказе от авторизации платежа.
- Для случаев использования большого коэффициента масштабирования, когда цифровая клавиатура не помещается целиком на экране, рядом с полями оплаты добавлен вывод отдельной цифровой клавиатуры. Для отображения клавиатуры необходимо удлиненное нажатие на поле оплаты (щелчок правой клавишей мыши).
- Применен новый алгоритм расчета цен позиций, который позволяет минимизировать количество разделений позиций на две при применении скидок. 
- Работа с незавершенными чеками. Усилен контроль незавершенных чеков с зафиксированной по ним оплатой через подключенные платежные системы. 
- В диалог ввода количества добавлен вывод наименования товара, для которого запрашивается количество. 

## Версия 4.6.2
### 15.08.2024
> **Дополнение профилей обмена с Фронтол 6 и xPOS 3 от 15.08.2024**
	• 
  Изменён алгоритм выгрузки и загрузки крепкого разливного алкоголя, учитываемого в литрах.
	- Такой алкоголь должен быть сопоставлен с алкокодом из ЕГАИС
	- Объем продукции должен быть указан равным 1 литру
	- Цена должна быть установлена за 1 литр.
	- При выгрузке такого алкоголя объем определяется из сопоставленного алкокода, а цена за бутылку пересчитывается с учетом полученного значения объема.
	- При обратной загрузке продаж количество проданного в миллилитрах пересчитывается в литры, а цена продажи пересчитывается обратно по значению объема из сопоставленного алкокода.
	• 
  Выгрузка и загрузка карточек крепкого разливного алкоголя, учитываемого в штуках/бутылках, осталась без изменений.
{.is-info}

### 06.07.2024
- **Обмен с оборудованием**. Доработан обмен с программным обеспечением Frontol 6 и Frontol xPOS 3 в части крепкого разливного алкоголя:
	- При выгрузке разливного алкоголя изменен порядок выгрузки объема продукции. Для алкоголя, учитываемого в литрах объем берется только из поля Объем карточки товара, а для учитываемого в штуках/бутылках объем определяется по алкокоду или из карточки товара, если не заполнено значение для алкокода.
	- При загрузке продаж для алкоголя, учитываемого в бутылках/штуках проданное количество в миллилитрах пересчитывается в штуки. Для алкоголя, учитываемого в литрах, проданное количество миллилитров пересчитывается в литры. 
	- Исправлена ситуация с загрузкой операций вскрытия как продажи. 
- **Работа с ФР**. Форматно-логический контроль, используемый для корректного пробития сумм и цен в кассовом чеке перенесен в ядро РМК и дополнительный вызов его из библиотек функций теперь не происходит. 
- **Маркировка**. В документе **Заказа кодов маркировки** для всех запросов получения остатков из ГИС МТ, теперь доступен выбор типа запрашиваемых упаковок в поле "Тип КМ". 
- **Схемы РМК**. Добавлена возможность указать интервал неактивности кассира, после которого РМК автоматически переходит в окно авторизации. 
- **Схемы РМК**. Добавлен признак "**Проверять наличие подарочной карты**". При установке признака и при вводе суммы оплаты **Сертификатом**, будет проверяться наличие подарочной карты в чеке. 
- **МРП 2**. Работа с разливными напитками. При закрытии чека теперь все позиции с разливными напитками переводятся в базовую единицу объема, по умолчанию – литры. 
Это позволяет не заводить объединяющие карточки товаров, учитываемые в литрах, а продавать алкоголь, разливая его непосредственно из бутылок, учитываемых в штуках. Ранее это было сделать затруднительно, т.к. не всегда имелся корректный пересчет из штук в нужное количество проданного напитка. Например, если объем бутылки 0.7 литра, то корректно пересчитать в штуки 50 мл не представляется возможным.
